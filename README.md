# Provenance Guard

A backend service that a creative-sharing platform can plug in to estimate whether a piece of
text-based content was written by a human or generated by AI and, crucially, to do so
**honestly about its own uncertainty**. Instead of forcing a binary verdict, Provenance Guard
returns a confidence score, surfaces a plain-language transparency label, and gives creators a
path to appeal a classification they believe is wrong.

Note: The flask backend is run on port 5001 for compatibility with MACOS.
---

## Quick start

```bash
python -m venv .venv
source .venv/bin/activate          # Mac/Linux
pip install -r requirements.txt

# .env in the repo root:
#   GROQ_API_KEY=your_key_here

python app.py                      # serves on http://127.0.0.1:5001
```

Then submit some text:

```bash
curl -s -X POST http://127.0.0.1:5001/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon, painting the sky in hues of amber and rose. I sat on the porch, coffee in hand, watching the neighborhood slowly go quiet.", "creator_id": "test-user-1"}' | python -m json.tool
```

## Architecture overview

```
SUBMISSION FLOW
  Creator
    │  POST /submit { text, creator_id }
    ▼
  Rate Limiter (Flask-Limiter)  ── over limit? → 429 (before any LLM cost)
    │ raw text
    ▼
  Detection Pipeline
    ├─ Signal 1: Groq LLM   → llm_score   (0–1, semantic)
    └─ Signal 2: Stylometry → stylo_score (0–1, structural)
    │ llm_score, stylo_score
    ▼
  Confidence Scorer
    ai_probability = 0.60·llm + 0.40·stylo
    confidence     = certainty · (1 − 0.5·directional_disagreement)
    asymmetric thresholds (high bar to say "AI")
    │ attribution, ai_probability, confidence
    ▼
  Label Generator → one of 3 variants (AI / human / uncertain) + confidence%
    │
    ├──► Audit Log (SQLite): content_id, creator_id, ts, attribution,
    │                        confidence, llm_score, stylo_score, status
    └──► HTTP response (JSON)

APPEAL FLOW
  Creator → POST /appeal { content_id, creator_reasoning }
    → look up original decision (404 if missing)
    → status → "under_review"
    → log appeal next to the original decision
    → confirmation response
  GET /log → most recent audit entries (reviewer queue + evidence)
```

**In words:** A submission to `POST /submit` first clears the rate limiter (so a flood of
requests can't run up LLM cost). The text then runs through **two independent detection
signals** — a Groq LLM that judges the text *semantically* and a pure-Python stylometric
analyzer that measures *structural* statistics. Their two scores are blended into an
`ai_probability` and a separate `confidence` value that is **drained when the signals
disagree**. A label generator turns that into one of three reader-facing labels, the whole
decision is written to a SQLite audit log, and the response comes back keyed by a
`content_id`. If a creator disputes the verdict, they send that `content_id` plus their
reasoning to `POST /appeal`, which flips the record's status to `under_review` and logs the
appeal alongside the original decision for a human reviewer to see via `GET /log`.

### API reference

| Endpoint  | Method | Accepts                              | Returns |
|-----------|--------|--------------------------------------|---------|
| `/submit` | POST   | `{text, creator_id}`                 | `{content_id, attribution, confidence, ai_probability, signals{…}, label}` |
| `/appeal` | POST   | `{content_id, creator_reasoning}`    | `{content_id, status:"under_review", message, original_decision}` |
| `/log`    | GET    | `?limit=N` (optional, default 50)    | `{entries:[…]}` newest-first audit rows |
| `/health` | GET    | —                                    | `{status:"ok"}` |

## Detection signals

The pipeline uses **two genuinely distinct signals** i.e. one *semantic*, one *structural*. They
capture different properties of the text and, importantly, **fail in different directions**, so
their agreement (or disagreement) is itself informative.

### Signal 1 — LLM classifier (Groq `llama-3.3-70b-versatile`) — *semantic*
- **What it measures:** A holistic "does this read like AI?" judgment — generic-but-fluent
  phrasing, even hedging, balanced both-sides framing, low risk-taking, smooth transitions,
  and the absence of an idiosyncratic human voice.
- **Why I chose it:** It reasons over *meaning and style together*, which no cheap heuristic
  can. It's the strongest single detector available on a free tier, so it carries the larger
  weight. Called at `temperature=0` for repeatability; returns
  `{"ai_score": 0–1, "rationale": "<sentence>"}`. If the API call fails or returns junk, the
  signal degrades to `0.5` (maximally uncertain) so a network blip never produces a confident
  accusation.
- **What it misses (blind spot):** It can be fooled by lightly-edited AI text (the obvious
  tells get smoothed away) and it can *over-flag* careful, formal, or non-native-English human
  writing as "AI-ish." It is also an external dependency and can be confidently wrong.

### Signal 2 — Stylometric heuristics — *structural*
Three measurable metrics, each mapped to a 0–1 "AI-likeness" contribution, then averaged into
`stylo_score`:
- **Sentence-length burstiness (variance):** humans swing between short punchy and long winding
  sentences; AI trends toward uniform medium-length sentences. *Low* variance → AI-like.
- **Type-token ratio (vocabulary diversity):** unique words ÷ total words. Treated as *neutral*
  for normal-length prose and only flags **genuinely repetitive** text (refrains, templated
  copy) as AI-like.
- **Punctuation & casing tidiness:** humans are irregular (lowercase sentence starts, "...",
  "?!", em-dashes, ALL-CAPS emphasis, contractions); AI is grammatically tidy. *High* tidiness
  → AI-like.
- **Why I chose it:** It's a structural lens that needs no API, no cost, and no understanding of
  meaning — and it's independent of Signal 1, so it can corroborate or contradict it.
- **What it misses (blind spot):** Short texts (< ~40 words) make variance and TTR statistically
  meaningless, so the signal is flagged **unreliable** and confidence is capped. Formal/academic
  human writing scores AI-like; casual or deliberately messy AI scores human-like; poetry and
  lists break the sentence-length assumption entirely.

**Why this pairing is more than the sum of its parts:** Signal 1 is fooled by *editing*; Signal
2 is fooled by *genre*. Because those failure modes don't overlap, agreement between them is
strong evidence and disagreement is an honest cue to back off to "uncertain."

## Confidence scoring

### How the two signals combine

```
ai_probability = 0.60 · llm_score + 0.40 · stylo_score
```

The LLM gets the larger weight because it's the stronger single detector; stylometry is the
cheaper, noisier structural check.

Confidence is computed **separately** from probability, and it measures *how sure we are of the
verdict*, not how high the probability is:

```
certainty             = |ai_probability − 0.5| · 2          # 0 at the fence, 1 at the extremes
raw_gap               = |llm_score − stylo_score|
directional_disagree  = raw_gap · (0.4 if both signals lean the same way else 1.0)
confidence            = certainty · (1 − 0.5 · directional_disagree)
# short / unreliable stylometry caps confidence at 0.5
```

The key idea is **directional disagreement**: two signals that *both* lean "AI" (or both lean
"human") at different intensities are corroborating each other, so they're only mildly
penalized. They're only fully penalized when they straddle 0.5 and point at *opposite* verdicts.
This is what lets a clearly-AI text reach a confident verdict while genuinely ambiguous text
collapses toward "uncertain."

### Thresholds — asymmetric on purpose

| Condition | Attribution | Label |
|---|---|---|
| `ai_probability ≥ 0.65` **and** `confidence ≥ 0.55` | `likely_ai` | Likely AI-generated |
| `ai_probability ≤ 0.40` **and** `confidence ≥ 0.50` | `likely_human` | Likely human-written |
| everything else | `uncertain` | Attribution uncertain |

The "AI" gate requires **both** a high probability **and** high confidence; the "human" gate is
easier to clear. When the system is unsure, it defaults to **uncertain** — it never falls back
to an AI accusation. `confidence ≈ 0.5` is the honest "we don't know" zone, which is exactly why
a `0.51` and a `0.95` result produce different labels.

### How I validated that the scores are meaningful

I ran a calibration set spanning clearly-AI, clearly-human, and deliberately borderline inputs
(including the project's provided samples) and checked three things: (a) clear cases land in the
right band with high confidence, (b) borderline cases land in/near *uncertain* with low
confidence, and (c) the scores **spread out** rather than clustering. They do — `ai_probability`
ranges from ~0.16 to ~0.85 across the set:

| Input | llm | stylo | → ai_probability | confidence | verdict |
|---|---|---|---|---|---|
| Strong AI (corporate marketing) | 0.90 | 0.77 | **0.850** | 0.68 | `likely_ai` |
| AI listicle | 0.90 | 0.74 | 0.835 | 0.65 | `likely_ai` |
| Casual human (ramen review) | 0.10 | 0.24 | **0.157** | 0.67 | `likely_human` |
| Borderline: formal human (monetary policy) | 0.70 | 0.63 | 0.670 | 0.34 | `uncertain` |
| Borderline: lightly-edited AI (remote work) | 0.40 | 0.64 | 0.495 | 0.01 | `uncertain` |
| Edge case: repetitive simple poem | 0.80 | 0.70 | 0.762 | 0.50 | `uncertain` |

The two borderline inputs and the poem all land in `uncertain` with low confidence — the
behavior the design intends.

### Two example submissions with noticeably different confidence

**High-confidence example** — strong AI corporate copy (`creator_id: writer_alice`):

```json
{
  "attribution": "likely_ai",
  "ai_probability": 0.8496,
  "confidence": 0.6817,
  "signals": { "llm_score": 0.9, "stylo_score": 0.7741 },
  "label": { "headline": "Likely AI-generated", "confidence_pct": 68 }
}
```

**Lower-confidence example** — lightly-edited "remote work" text where the two signals split
(LLM leans human at 0.40, stylometry leans AI at 0.64), so confidence collapses
(`creator_id: writer_bob`):

```json
{
  "attribution": "uncertain",
  "ai_probability": 0.4952,
  "confidence": 0.0085,
  "signals": { "llm_score": 0.4, "stylo_score": 0.6379 },
  "label": { "headline": "Attribution uncertain", "confidence_pct": 1 }
}
```

Confidence drops from **0.68 → 0.01** between the two — concrete evidence the score reflects
genuine uncertainty rather than being a constant.

## Transparency label

The label is the reader-facing output. It's written in plain language, always states a
confidence percentage, and the AI variant is deliberately **hedged** (`likely`, `estimate`,
`can appeal`) so a possible false positive doesn't read as a verdict. `{confidence_pct}` =
`round(confidence × 100)`.

**Variant A — High-confidence AI** (`attribution == likely_ai`):

> **Likely AI-generated**
> Our analysis suggests this text was probably produced with significant AI assistance
> (confidence: {confidence_pct}%). This is an automated estimate, not a certainty and the
> creator can appeal if this is wrong.

**Variant B — High-confidence human** (`attribution == likely_human`):

> **Likely human-written**
> Our analysis found no strong signs of AI generation in this text (confidence:
> {confidence_pct}%). This is an automated estimate, not a guarantee.

**Variant C — Uncertain** (`attribution == uncertain`):

> **Attribution uncertain**
> Our signals disagree or are inconclusive for this text, so we are not assigning an
> attribution (confidence: {confidence_pct}%). When we can't tell, we don't label and the
> benefit of the doubt goes to the creator.

All three variants are reachable in practice.

## Appeals workflow

- **Who can appeal:** any creator holding a `content_id` from a prior `/submit` response.
- **What they provide:** `content_id` + free-text `creator_reasoning`.
- **What the system does:** looks up the original decision (404 if not found), sets that
  content's **status → `under_review`**, writes a new `appeal` audit entry storing the
  reasoning *alongside* a snapshot of the original decision, and returns a confirmation. There
  is **no automated re-classification** — the item simply enters a human reviewer's queue.

```bash
curl -s -X POST http://127.0.0.1:5001/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id":"PASTE-CONTENT-ID","creator_reasoning":"I wrote this myself; I am a non-native English speaker and my style is formal."}' | python -m json.tool
```

## Rate limiting

Applied to `POST /submit` via Flask-Limiter (in-memory storage), keyed per client IP:

```
10 per minute ; 60 per hour ; 200 per day
```

**Reasoning — these numbers model a real creative platform, not arbitrary caps:**
- **10/minute** — a genuine creator submits their own work a handful of times; nobody legitimately
  needs more than ten submissions in a single minute. This is the burst cap that stops a script
  from hammering the endpoint. It matters because every `/submit` triggers a *paid* LLM call, so
  the per-minute cap is also direct cost control.
- **60/hour** — comfortably above a heavy-but-real editing session (resubmitting drafts), while
  bounding sustained automated abuse from a single source.
- **200/day** — an upper sanity bound on one IP's daily volume; a real writer is nowhere near it,
  but it caps a slow-drip scraper.

**Evidence (12 rapid requests, limit 10/min):**

```
request 1  -> HTTP 200
request 2  -> HTTP 200
...
request 10 -> HTTP 200
request 11 -> HTTP 429
request 12 -> HTTP 429
```

The 429 response body:

```json
{
  "error": "rate_limit_exceeded",
  "message": "Too many requests — 10 per 1 minute. Please slow down and try again later."
}
```

## Known limitations

The system will most reliably misclassify the **non-native-English speaker writing in a careful,
formal register**. Such writing is grammatically tidy and evenly paced, which pushes *both*
signals toward "AI": the stylometric analyzer reads the uniformity and low irregularity as
machine-like, and the LLM reads the polished, generic-feeling fluency the same way. Because the
two signals *agree* (in the wrong direction), confidence stays high and the text can be labeled
`likely_ai` — a false positive against exactly the kind of creator least able to argue back.
This is a property of *what the signals measure*, not a lack of data: "uniform and tidy" is
genuinely what both detectors key on. The mitigations are structural — the asymmetric "AI"
threshold, the deliberately hedged label wording, and the appeals workflow (the
`writer_alice` example above is precisely this case, and it's why that appeal exists).

A second known weak spot is the **repetitive, simple-vocabulary poem**: low sentence-length
variance and low lexical diversity look "AI-like" to the stylometric signal. Here the design
holds up — it lands in `uncertain` rather than a false accusation — but it illustrates that the
structural signal has no concept of poetic intent.

## Spec reflection

**One way the spec helped.** Writing the three label variants and the threshold table in
`planning.md` *before* any code meant I was implementing against concrete targets instead of
inventing them mid-stream. When I built `labels.py` and `scoring.py`, the work was mechanical —
the hard decisions (what does 0.5 confidence mean? what's the exact wording of the AI label?)
were already made, and the code just had to match them. Deciding "what 0.5 means to a user"
first, then building the math to achieve it, is straight from the spec's guidance and it kept
the confidence model coherent.

**One way the implementation diverged.** My spec's confidence formula used raw magnitude
disagreement (`|llm − stylo|`), and its TTR metric mapped diverse vocabulary straight to 0. In
calibration both choices misbehaved: the magnitude penalty made `likely_ai` essentially
unreachable (two signals both *correctly* leaning AI at different intensities were treated as
conflicting), and the TTR mapping pinned itself to 0 for all normal-length prose, capping the
achievable `stylo_score`. I diverged by (1) making disagreement **directional** only fully
penalizing signals that point at opposite verdicts and (2) recalibrating TTR to be **neutral**
for normal prose and reactive only to genuine repetition. Both changes came from watching real
scores rather than reasoning in the abstract, which is exactly the gap testing is supposed to
surface.

## AI usage

**Instance 1 — fixing an unreachable verdict.** During Milestone 4 calibration, no clearly-AI
text could reach the `likely_ai` band; everything collapsed to `uncertain`. I directed the AI
assistant to diagnose why, and it identified that my confidence formula penalized *magnitude*
disagreement even when both signals agreed on *direction* (both > 0.5). I accepted the diagnosis
but **revised the proposed fix**: rather than just lowering the threshold (which would have made
the poem edge case a false positive), I had it implement *directional* disagreement a mild
penalty when signals corroborate, a full penalty only when they conflict. I verified the change
against the whole calibration set to confirm the poem still landed `uncertain`.

**Instance 2 — recalibrating a dead metric.** I asked the AI to generate the stylometric signal
from my spec. Its first TTR mapping (threshold 0.75) looked reasonable but, on testing, returned
0 for *every* normal-length sample — AI and human alike — so it never discriminated and dragged
the AI ceiling down. I directed a recalibration so TTR is neutral (~0.5) for ordinary prose and
only spikes for genuinely repetitive text, then re-ran the calibration table to confirm the
`stylo_score` spread widened and `likely_ai` became reachable. I also overrode an earlier
ALL-CAPS heuristic that was counting the acronym "AI" as human-style emphasis.
